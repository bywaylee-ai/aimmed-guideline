# BizCare 화면설계서 작성 하네스 (피그마 연동용)

## 목적
기능정의서(기능 정의 시트)를 입력받아 피그마 연동 가능한 구조의 화면설계서를 생성한다.

---

## 입력 형식

작업 요청 시 다음 정보를 제공한다:

```
[화면 ID] eap-login-01
[화면명] 로그인
[화면 종류] Page
[표시내용] (기능정의서 K열 내용 붙여넣기)
[내용] (기능정의서 L열 내용 붙여넣기)
[비고] (기능정의서 M열 내용 붙여넣기)
```

---

## 산출물 구조 (피그마 연동 기준)

### 화면설계서 1건 = 아래 5개 섹션으로 구성

---

### 섹션 1. 화면 기본 정보

```
화면 ID      : eap-login-01
화면명       : 로그인
화면 종류    : Page
분류         : 로그인
연결 화면    : (진입 전 화면) → 로그인 → (이후 분기 화면들)
```

---

### 섹션 2. 레이아웃 구조 (피그마 Frame 기준)

- 최상위 Frame 이름 = 화면명
- 내부 구성요소를 **Zone** 단위로 구분

```
[Zone A] Header
[Zone B] Content Area
  ├── [Component 1] 이름 입력폼
  ├── [Component 2] 비밀번호 입력폼
  └── [Component 3] CTA 버튼
[Zone C] Footer / 헬퍼 텍스트
```

- 표시내용(K열)의 번호 순서대로 위→아래 배치
- 블럭(블럭형 화면 종류)은 독립 Frame이 아닌 Component로 표기

---

### 섹션 3. 컴포넌트 명세

표시내용 번호별로 아래 형식 작성:

```
번호  | 컴포넌트 유형 | 컴포넌트명          | 상태            | 비고
------|--------------|--------------------|-----------------|---------
1-1   | Input        | 이름 입력폼         | Default / Error  |
1-2   | Input        | 비밀번호 입력폼     | Default / Error  | 마스킹 처리
1-3   | Button       | 로그인 CTA 버튼    | Default / Active |
2     | Helper Text  | 초기 비밀번호 안내  | Static           |
3     | Link Text    | 비밀번호 찾기       | Default / Hover  |
```

**컴포넌트 유형 매핑** (표시내용 [대괄호] → 피그마 컴포넌트):

| 표시내용 태그 | 피그마 컴포넌트 유형 |
|---|---|
| [입력폼] | Input Field |
| [CTA 버튼] | Button / Primary |
| [버튼] | Button / Secondary |
| [헬퍼 텍스트] | Helper Text |
| [링크] | Link Text |
| [리스트] | List |
| [카드] | Card |
| [드롭다운] | Dropdown / Select |
| [배너] | Banner |
| [안내 문구] | Label / Body Text |
| [프로세스 스텝] | Stepper |
| [블럭] | Section Component |
| [바텀시트] | Bottom Sheet |
| [모달] | Modal / Dialog |
| [뱃지] | Badge / Chip |
| [토글] | Toggle |
| [탭] | Tab Bar |
| [캘린더] | Calendar |
| [체크박스] | Checkbox |
| [라디오 버튼] | Radio Button |
| [아코디언] | Accordion |

---

### 섹션 4. 인터랙션 & 분기 흐름

내용(L열)의 동작/분기를 화살표 형식으로 표현:

```
[트리거]        [조건]                    [결과]
로그인 버튼 탭  입력값 일치               → 본인 확인(간편인증) 화면 이동
               비밀번호 강제 변경 조건    → 비밀번호 강제 변경 화면 이동
               입력값 불일치             → 인라인 에러 메시지 표시
비밀번호 찾기  -                         → 비밀번호 찾기 화면 이동
```

---

### 섹션 5. 검토 사항 (비고 기반)

비고(M열) 내용을 아래 형식으로 정리:

```
유형        | 내용                              | 담당
-----------|-----------------------------------|-------
정책 확인   | 어드민 기업/대상자 등록 프로세스    | EAP팀
개발 확인   | PASS 인증 웹뷰 연동 방식           | 개발팀
디자인 확인 | 초기 비밀번호 안내 문구 위치        | 디자인팀
미확정      | 카카오·토스 인증 제공 여부          | -
```

---

## 실행 절차 (하네스)

1. **입력 파싱**: 화면 ID, 화면명, 표시내용, 내용, 비고 추출
2. **섹션 1** 기본 정보 작성
3. **섹션 2** 표시내용 번호 순서대로 Zone/Component 구조화
4. **섹션 3** 컴포넌트 유형 매핑 (위 매핑 테이블 기준)
5. **섹션 4** 내용(L열)에서 트리거-조건-결과 추출
6. **섹션 5** 비고에서 유형 분류 (정책/개발/디자인/미확정)
7. **검증**: 표시내용 번호 ↔ 내용 번호 동기화 확인

---

## 복수 화면 일괄 작업 시

- 화면 ID 목록을 제공하면 순서대로 처리
- 각 화면 사이 구분선(`---`) 삽입
- 연결 화면 관계 자동 추론 (기능정의서 내용 기반)

---

## 피그마 전달 형식 규칙

- Frame 이름: `{화면 종류}/{화면명}` (예: `Page/로그인`)
- Component 이름: `{유형}/{상태}` (예: `Button/Primary/Default`)
- 색상 토큰: 디자인팀 Design System 기준 따름 (별도 정의 시 업데이트)
- 폰트: 디자인팀 기준 따름

---

## 주의사항

- 블럭(블럭형)은 독립 Page가 아닌 상위 Page 내 Component로 처리
- 비고에 `재검토`, `확인 필요` 키워드 있는 항목은 섹션 5에 반드시 포함
- 화면 종류가 `버튼`인 경우 독립 화면설계서 생성하지 않고 상위 화면 섹션 3에 통합
